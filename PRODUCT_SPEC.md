# 国产历史剧全景时间轴 — 产品概念文档

## 一、产品定位

### 一句话定义

一个以"时间/朝代"为绝对锚点的国产历史剧可视化索引引擎，具备顶级 Web 交互质感。

### 核心价值

- **表层价值（用户感知）：** 极致视觉秩序感的历史剧地图。没有广告、没有算法推荐、没有信息噪音，只有一张克制、高级的历史画卷。
- **底层壁垒（长期竞争力）：** 一套结构化的历史剧知识图谱——将剧集与真实历史的时间、事件、人物精确关联。

### 知识图谱构建路径

分三个阶段逐步深化：

| 阶段 | 关联维度 | 说明 |
|------|---------|------|
| Phase 1 | **时间** | 建立剧集与年份、朝代的映射关系，完成时间轴核心体验 |
| Phase 2 | **事件** | 在时间轴上叠加历史事件节点（玄武门之变、靖难之役……），将剧集锚定到具体事件 |
| Phase 3 | **人物** | 引入历史人物图谱，建立人物-剧集-事件的三角关联网络 |

---

## 二、目标用户

### 考据型用户

> "唐朝中后期有哪些剧？和真实历史对得上吗？"

- 按朝代/事件主动浏览
- 高度依赖分类滤镜
- 核心诉求：信息密度、历史准确性

### 选剧型用户

> "我刚看完《大明风华》很上头，这个时期还有什么类似的？"

- 从一部剧出发，发现时间轴上的相邻剧集
- 核心诉求：发现效率、快速决策

两类用户的需求通过功能同时满足，不做优先级区分。

---

## 三、核心痛点

1. **历史时间坐标错乱：** 观众看完《大明风华》看《山河月明》，脑子里缺乏"朱元璋→朱棣→朱祁镇"的绝对坐标系，难以产生历史维度的共鸣。

2. **正史与演义的维度污染：** 考据党和爽剧受众互相觉得对方吵闹。把《康熙王朝》和《戏说乾隆》混在一个列表里，对双方都是噪音。

3. **长尾检索效率极低：** 想看"五代十国"或特定历史事件（如靖难之役）相关的剧，搜索引擎给出的全是营销号的碎片盘点。

---

## 四、分类体系

采用单一维度的二元分类，简洁明确：

| 分类 | 定义 | 示例 |
|------|------|------|
| **严肃正剧** | 主要人物、事件高度还原正史，以历史真实为创作基底 | 《大明王朝1566》《觉醒年代》《贞观之治》 |
| **史事演义** | 以真实朝代/事件为背景进行文学化改编，人物和情节有显著虚构成分 | 《琅琊榜》《知否》《庆余年》《长安十二时辰》 |

全局提供一个拨片开关，用户可一键切换视界或同时显示两类。

> **设计原则：** 分类仅此两个，不做更多细分。模糊地带由数据录入时人工判定。宁可武断，不可暧昧。

## 五、核心功能（MVP）

### 功能 1：可折叠的手风琴时间轴

**设计意图：** 彻底抛弃绝对等比年份映射。以"朝代"为一级折叠单元，"核心帝王/纪元"为二级单元。解决清朝剧集扎堆爆炸、魏晋南北朝大片留白的"密度灾难"。

**交互规格：**

- **主轴方向：** 水平方向，从左（远古）到右（现当代），符合时间线的天然心智模型。
- **一级节点（朝代）：** 折叠态下只显示朝代名称和年份跨度，点击展开进入二级。
- **二级节点（纪元/帝王）：** 展开后以紧凑网格排布该朝代下的剧集节点。
- **折叠/展开动画：** 使用 `View Transitions API` + CSS `spring()` 缓动，追求物理弹性感，交互反馈干脆无拖泥带水。

**并行政权处理（关键设计挑战）：**

三国、南北朝、五代十国等存在并行政权的时期，采用**分叉轨道**设计：

```
──── 东汉 ────┬── 魏 ──────────────┐
              ├── 蜀 ──────────┐   ├── 西晋 ────
              └── 吴 ──────────┘   │
                                   │
```

- 主时间轴在并行政权段分叉为多条平行轨道，每条轨道承载对应政权的剧集。
- 分叉与合流的节点配有历史事件标记（如"赤壁之战"分叉、"三家归晋"合流）。
- 使用 SVG 路径 + `motion` 动画库绘制流畅的分叉/合流曲线。

**跨朝代剧集处理：**

横跨多个朝代的剧集（如《大秦赋》覆盖战国末→秦统一，《楚汉传奇》覆盖秦末→汉初）：

- 在时间轴上渲染为一条**跨度条 (Span Bar)**，起止点锚定到对应的时间区间。
- 跨度条半透明，悬停时高亮并显示完整时间跨度。
- 同时出现在涉及的每个朝代的展开视图中（标记为"跨朝代"）。

**同期剧集密集排布：**

三国、清宫等"剧集核爆区"，在二级展开后采用**蜂巢网格布局 (Hexagonal Grid)**：

- 每部剧集为一个六边形节点，尺寸统一。
- 评分通过节点边框颜色/粗细编码（金色 = 9+，银色 = 8+，灰色 = 8 以下）。
- 鼠标悬停触发悬浮卡片，节点本身只显示剧名缩写。

**视觉风格：**

- 底色：深色主题（近黑的暖灰 `#1a1a1e`），配合低饱和度的朝代配色系统。
- 字体：中文使用思源宋体（Noto Serif SC）作为标题字体，营造历史卷轴感；正文使用思源黑体（Noto Sans SC）保证可读性。
- 时间轴主线：使用纤细的金色线条（`#c9a96e`），呼应古籍装帧质感。
- 整体气质对标：博物馆数字展厅的克制感，不是大屏数据可视化的炫技感。

---

### 功能 2：考据滤镜

**交互形式：** 全局顶部固定的极简切换组件，两个状态按钮：

```
[ 严肃正剧 | 史事演义 ]
```

- 默认两个都选中（全部显示）。
- 点击其一则只显示对应分类，时间轴上的节点以 `layout animation` 平滑重排。
- 滤镜切换时不重新加载页面，节点以淡入/淡出 + 位移动画过渡。

---

### 功能 3：剧集悬浮卡片

鼠标悬停剧集节点后弹出，信息结构如下：

| 字段 | 说明 | 示例 |
|------|------|------|
| **剧名** | 全称 | 大明王朝1566 |
| **评分** | 豆瓣评分 + 评分人数 | 9.7（12.8万人） |
| **时间锚点** | 剧集对应的核心历史时段 | 明嘉靖三十九年—四十五年（1560-1566） |
| **核心张力** | 一句话点出戏剧核心矛盾 | "嘉靖、严嵩、海瑞的三角权力绞杀" |
| **集数 · 年份** | 快速判断观看成本 | 46集 · 2007年 |
| **在哪看** | 当前可用的播放平台（可点击跳转） | 优酷 · 腾讯视频 |

**交互规格：**

- 悬停 200ms 延迟后弹出，避免鼠标快速划过时闪烁。
- 卡片使用 `backdrop-filter: blur()` 毛玻璃背景，配合微妙的投影。
- 卡片定位自动避让视口边缘。
- 点击卡片进入剧集详情页（Phase 2+，MVP 阶段卡片即终点）。

## 六、数据库结构

### 核心实体

#### 1. 朝代 (Dynasty)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 唯一标识，如 `tang`, `ming` |
| name | string | 朝代名称，如 "唐朝" |
| start_year | integer | 起始年份（负数表示公元前） |
| end_year | integer | 结束年份 |
| display_order | integer | 显示排序（处理并行政权时多个朝代可能有相同的时间范围但不同的排序） |
| parent_id | string \| null | 父级朝代 ID，用于处理并行政权（如"魏"的 parent 是"三国"） |
| color | string | 该朝代的主题色 hex 值 |

#### 2. 剧集 (Drama)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 唯一标识 |
| title | string | 剧名 |
| category | enum | `serious` \| `romance`（严肃正剧 / 史事演义） |
| douban_rating | float \| null | 豆瓣评分 |
| douban_rating_count | integer \| null | 豆瓣评分人数 |
| episode_count | integer | 集数 |
| release_year | integer | 首播年份 |
| story_start_year | integer | 剧情起始年份（历史时间） |
| story_end_year | integer | 剧情结束年份（历史时间） |
| dynasty_ids | string[] | 关联朝代 ID 列表（支持跨朝代） |
| historical_anchor | string | 历史时间锚点描述文本 |
| core_tension | string | 一句话核心张力 |
| poster_url | string \| null | 海报图片 URL |
| platforms | Platform[] | 播放平台列表 |

#### 3. 播放平台 (Platform) — 嵌套在 Drama 中

| 字段 | 类型 | 说明 |
|------|------|------|
| name | string | 平台名称，如 "优酷", "腾讯视频" |
| url | string | 该剧在该平台的播放链接 |

#### 4. 历史事件 (HistoricalEvent) — Phase 2

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 唯一标识 |
| name | string | 事件名称，如 "玄武门之变" |
| year | integer | 发生年份 |
| dynasty_id | string | 所属朝代 |
| description | string | 简要描述 |
| related_drama_ids | string[] | 关联剧集 ID 列表 |

#### 5. 历史人物 (HistoricalFigure) — Phase 3

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 唯一标识 |
| name | string | 人物名称 |
| birth_year | integer \| null | 出生年份 |
| death_year | integer \| null | 去世年份 |
| dynasty_id | string | 主要活动朝代 |
| description | string | 简要描述 |
| related_drama_ids | string[] | 出现在哪些剧集中 |
| related_event_ids | string[] | 参与的历史事件 |

### 数据存储方案

MVP 阶段使用本地 JSON 文件作为数据源，按实体类型组织：

```
data/
├── dynasties.json      # 朝代数据
├── dramas.json         # 剧集数据
└── events.json         # Phase 2: 历史事件数据（预留）
```

不引入数据库，全量 JSON 在构建时静态加载。数据规模（百级剧集）完全不需要后端。

## 七、技术选型方向

| 层面 | 选型思路 |
|------|---------|
| 框架 | 现代前端框架（React/Vue/Svelte 均可），重点是动画能力 |
| 动画 | Framer Motion / GSAP / View Transitions API，追求物理弹性感 |
| 时间轴渲染 | SVG + Canvas 混合方案：主轴用 SVG（可交互），密集节点区用 Canvas（性能） |
| 样式 | Tailwind CSS / CSS Modules，配合自定义的朝代配色 Design Token |
| 部署 | 静态站点（Vercel / Cloudflare Pages），零服务器成本 |
| 数据 | 本地 JSON 文件，构建时注入 |

## 八、MVP 范围与边界

### 做什么

- 覆盖 **8 个核心朝代**：秦、汉、三国、唐、宋、元、明、清
- 录入 **100-150 部** 代表性剧集
- 完成手风琴时间轴的折叠/展开交互
- 完成考据滤镜的切换
- 完成悬浮卡片的信息展示
- 并行政权的分叉轨道（至少覆盖三国）
- 跨朝代剧集的跨度条展示

### 不做什么

- 不做移动端适配
- 不做用户系统 / 登录
- 不做评论 / UGC
- 不做搜索（Phase 2 考虑）
- 不做 SEO 静态页面生成（Phase 2 考虑）
- 不做数据统计（后期考虑）
- 不考虑商业化

---

## 九、预期产品效果

1. **第一眼的质感胜利：** 打开网站，没有广告位，没有侧边栏杂音，只有一条从秦到清的克制历史画卷，深色背景上金色时间线蜿蜒展开。

2. **发现的效率革命：** 展开"明朝"，所有明朝相关剧集以蜂巢网格铺开，悬停即可获得核心判断信息，从"想看明朝的剧"到"决定看哪一部"，路径缩短到 10 秒以内。

3. **自传播的情绪价值：** 这张"历史剧地图"本身就是社交货币。考据党用它论战，剧迷用它推剧，历史爱好者用它当索引。一个填补了绝对垂直 Niche 的工具型产品。
